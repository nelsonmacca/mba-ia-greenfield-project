## Resolved Issues

- **Post-review fix (`fix/phase-03-e2e-teardown`) — full `npm run test:e2e` teardown.** Professor review found the complete e2e suite intermittently red even though each suite passed in isolation: `test/videos.e2e-spec.ts` enqueues **real** jobs onto the shared `video-processing` queue, which the `video-worker` consumes asynchronously (writing to the shared test DB), and its `afterAll` only did `queue.obliterate({ force: true })` + `app.close()`. That left in-flight/retry-delayed jobs and a leaked BullMQ Redis connection alive past the suite; the next suite's FK-ordered `cleanAllTables` (e.g. `auth.e2e`) then collided → foreign-key / cross-suite contamination. **Fix (teardown only, no product/endpoint/migration/Docker change):** the videos `afterAll` now `queue.pause()` → `queue.obliterate({ force: true })` (drops waiting **+ delayed retries + active**) → `queue.close()` (BullMQ `obliterate` does not release the connection; `app.close()` alone left it dangling) → `cleanAllTables(dataSource)` (leave the shared DB empty for the next suite) → `app.close()`. Validations: `npm run lint` and `npx tsc --noEmit` exit 0; full unit/integration `npm test -- --runInBand` green (217 passed, 3 skipped — the worker-only FFmpeg integration); full `npm run test:e2e -- --runInBand` green (72/72) and **stable across 4 consecutive runs**. Operational note still applies: after `docker compose down && up` on a fresh volume, run `npm run migration:run` before the e2e (else `relation "videos" does not exist`).
- **SI-03.7 (Playback & download — presigned GET, streaming) implemented.** `GET /videos/:id/playback` is **public** (`@Public()`) and `GET /videos/:id/download` is **authenticated** (no per-owner check — finer-grained authz for private/unlisted is Phase 04+, per the documented plan). `VideosService.getPlaybackUrl`/`getDownloadUrl` both load the video (404 `VIDEO_NOT_FOUND`), require `status = ready` (else 409 `VIDEO_NOT_READY` — a `ready` video lacking `object_key` is also treated as not-ready), and issue a short-lived presigned GET for the source object via the existing `StorageService.getPresignedDownloadUrl` (TD-06). Download adds `Content-Disposition: attachment; filename="{videoId}"`. Bytes never pass through the API — MinIO/S3 serves them and honours HTTP Range; the e2e asserts a `Range` request against the issued playback URL returns **206 Partial Content**. New `VideoUrlResponseDto` (`{ url }`) and `VideoNotReadyException` (`VIDEO_NOT_READY`/409). No new libraries. Tests: 27 unit (`videos.service.spec.ts`) + new e2e cases (playback public + Range→206, 409 before ready, 404 unknown; download 200 authed, 401 unauth, 409 before ready). `tsc --noEmit` and project-wide `npm run lint` exit 0. HLS/transcoding and API-proxied delivery remain out of scope (deferred). This completes SI-03.1..03.7; SI-03.8 (final validation + docs) remains.
- **SI-03.6 (Video status & metadata read endpoint) implemented.** `GET /videos/:id` is **public** (`@Public()`) per the product rule that anonymous users can watch and the documented SI-03.6 contract. `VideosService.getById` loads the video (404 `VIDEO_NOT_FOUND` when absent) and returns the safe, enxuto shape `{ id, status, title, duration_seconds?, thumbnail_url?, created_at }` via the new `VideoResponseDto`. Internal storage keys (`object_key`/`thumbnail_key`) and processing internals (`processing_error`/`size_bytes`/`content_type`) are **never** exposed. `duration_seconds`/`thumbnail_url` are omitted (key absent) until processing produces them; the `thumbnail_url` is a short-lived presigned GET issued via the existing `StorageService.getPresignedDownloadUrl` (TD-06) only when `thumbnail_key` is set. No new libraries. Tests: 16 unit (`videos.service.spec.ts`, incl. the three `getById` cases) + 3 new e2e (`videos.e2e-spec.ts`: public read without auth, 404, 400 malformed uuid); the full `videos.e2e-spec.ts` suite is green (**14/14**, ~59s). `tsc --noEmit` and project-wide `npm run lint` exit 0. The e2e `beforeAll` was given an explicit `60000ms` timeout because the full `AppModule` bootstrap (Postgres + Redis + MinIO + Throttler) can exceed Jest's default 5000ms hook timeout on a loaded host. **Operational note:** after a `docker compose down && up` the local DB must be re-migrated (`npm run migration:run`) before running the e2e — otherwise it fails with `relation "videos" does not exist`. Authentication/ownership and the richer/internal payload were deliberately **not** added (out of the documented public contract; private/unlisted authz is Phase 04+). Playback/download (SI-03.7) remains out of scope.
- **SI-03.5 (Video worker — BullMQ consumer + FFmpeg/ffprobe) implemented.** `VideoProcessor` (`@Processor(video-processing)` `WorkerHost`) consumes `process-video` jobs and delegates to `VideoProcessingService`, which: loads the video, short-circuits if already `ready` (idempotent re-delivery), sets `processing`, streams the source object from MinIO to a temp file (`StorageService.downloadToFile` — worker-only byte handling, never the API), extracts duration via ffprobe (`FfmpegService.probeDurationSeconds`), generates a single-frame thumbnail (`generateThumbnail`, 50%, 640px) and uploads it to `videos/{id}/thumbnail.jpg`, then persists `duration_seconds`/`thumbnail_key`/`status=ready`. On error it rethrows so BullMQ retries (`attempts:3`), marking `failed`+`processing_error` only on the final attempt; temp dir cleaned in `finally`. **TD-04 gating:** the consumer is registered only when `WORKER_MODE=true` (`workerOnlyProviders`), so the API process never consumes jobs — asserted in `videos.module.spec.ts`. `fluent-ffmpeg`+`@types/fluent-ffmpeg` installed at pinned versions with explicit authorization; context7 ffprobe/screenshots + BullMQ WorkerHost APIs re-verified. Unit suite (7) + module gate green in `nestjs-api`; the real-DB+MinIO+FFmpeg integration (3/3) runs in the `video-worker` container (auto-skips with a warning where ffmpeg is absent — see advisory). `tsc --noEmit` and project-wide `npm run lint` exit 0. Streaming/download (SI-03.7) and the status read endpoint (SI-03.6) remain out of scope.
- **SI-03.4 (Upload confirmation + job publishing) implemented.** `QueueModule` registers BullMQ over the provisioned Redis (`forRootAsync` from `queueConfig`) and the `video-processing` queue (`attempts:3`, exponential backoff, `removeOnComplete`, `removeOnFail:false`). `VideoProducerService.enqueueProcessing` publishes `process-video { videoId, objectKey }` — payload only, never bytes (TD-03). `POST /videos/:id/confirm` (authenticated, owner-only) validates the video exists (404 `VIDEO_NOT_FOUND`), belongs to the user's channel (403 `FORBIDDEN_VIDEO_ACCESS`), and that the object is present in MinIO via `objectExists`/`headObject` (409 `UPLOAD_NOT_CONFIRMED`); it re-reads and re-validates `size_bytes`/`content_type` from `HeadObject` (≤10GB), transitions `draft → uploaded → queued`, and enqueues exactly one job. Re-confirm is idempotent (no duplicate job). Tests use **real Redis** and assert the enqueued job via `queue.getJobs`; isolation via `queue.obliterate` per test. 31 unit/integration + 11 e2e green; `tsc` clean; SI-03.4 files lint-clean. The `@nestjs/bullmq` + `bullmq` libs were installed at pinned versions with explicit authorization; context7 BullMQ docs re-verified at implementation. The worker consumer, FFmpeg, thumbnail, and streaming/download remain out of scope (SI-03.5+). A pre-existing project-wide lint failure was discovered during validation and recorded as an advisory (deferred to a separate task per user direction).
- **SI-03.3 (Storage module + presigned upload / draft creation) implemented.** `StorageModule`/`StorageService` wrap a configured `S3Client` (MinIO endpoint, `forcePathStyle`, credentials from `storageConfig`) and expose `getPresignedUploadUrl` (PUT), `getPresignedDownloadUrl` (GET), `headObject`, and `objectExists`. `POST /videos` (authenticated) pre-registers a `draft` `Video` for the user's channel, derives `object_key = videos/{id}/source` (TD-05), issues a presigned PUT URL (TD-01), and returns `{ id, upload_url, object_key, status }`. The file never passes through the API. `FileTooLargeException` (`FILE_TOO_LARGE` / 400) added; `size_bytes`/`content_type` are client-asserted now and re-validated on confirm (SI-03.4). The `@aws-sdk/client-s3` + `@aws-sdk/s3-request-presigner` libraries (absent until now) were installed at the pinned versions with explicit authorization. context7 docs for the AWS SDK were re-verified at implementation against the installed version (presigned PUT/GET + HeadObject APIs match `library-refs.md`). Tests: 26 unit/integration (including a real-MinIO PUT→GET round-trip and a real-DB+MinIO draft path) + 5 e2e, all green; `tsc --noEmit` and `lint` exit 0; all 3 migrations apply on a fresh DB.
- **SI-03.2 (Video entity + migration) implemented.** The `Video` entity (linked to `Channel` via `channel_id` FK, `onDelete: CASCADE`), the `video_status` enum (`draft/uploaded/queued/processing/ready/failed`, default `draft`), `(channel_id)` + `(status)` indexes, and the CLI-generated `CreateVideos` migration are in place and verified against a fresh DB. The `migrations.integration-spec.ts` advisory pattern (drop the enum type explicitly — the `bb0010e` lesson) was extended to `video_status`. A Postgres deadlock surfaced when `videos` joined the parallel `DROP TABLE ... CASCADE` cleanup; resolved by making the drops sequential. No new libraries (TypeORM-only; UUID PK per TD-05). Full suite (162 tests) + `tsc --noEmit` green.
